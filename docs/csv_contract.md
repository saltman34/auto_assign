# CSV and data contracts

Single reference for **file formats** the app parses or exports. Implementation: `src/auto_assign/core/csv_parsing/`, `parse_tech_profiles.py`, `ingestion/csv_schema.py`.

For **how profiles land in the database** and uniqueness rules, see [technician_profiles_ingestion.md](technician_profiles_ingestion.md).

---

## 1. Schedule CSV (input)

**Purpose:** One row per technician per calendar day with shift availability.

### Required columns (after header normalization)

| Column | Type / notes |
|--------|----------------|
| `tech_name` | Non-empty string. Normalized to **title case** for matching. Must match a **`tech_name`** stored in **Technician Profiles** (so the app can resolve `tech_id`). |
| `date` | `YYYY-MM-DD`. Parsed as a calendar date (`work_date`). |
| `available_AM` | Boolean-like: `yes`/`no`, `true`/`false`, `1`/`0`, etc. See `normalize_schedule.py`. |
| `available_MID` | Same as above. |
| `available_PM` | Same as above. |
| `staffing_status` | One of: `scheduled`, `call_off`, `overtime` (aliases: spaces/underscores, case-insensitive). See `Staffing_Status` in `domain/enums.py`. |

**Empty cells** in any required column are rejected.

### Legacy header aliases

`standardize_schedule_column_names` may rename:

- `availableAM` â†’ `available_AM` (and MID/PM) if snake_case columns are absent  
- `staff_status` â†’ `staffing_status`

### Pool rule used by the engine

For a chosen **date** and **time slot**:

- Rows must match that `date`.
- `staffing_status == call_off` â†’ technician is **not** in the pool for **any** slot that day (`get_available_techs.py`).
- Otherwise, the technician is in the pool only if the availability flag for that slot is **true** (e.g. AM â†’ `available_AM`).

## 2. Technician profile CSV (input)

**Purpose:** Upsert technicians: stable `tech_id`, display `tech_name`, preferences.

### Required columns

| Column | Notes |
|--------|--------|
| `tech_id` | Stable id; normalized (see `normalize_tech_id` in primitives). |
| `tech_name` | Display / schedule name; must be **unique** in DB (`uq_technicians_tech_name`). |
| `daily_preference` | `consistency` or `variation` (see `DailyPreference`). |

### Optional columns

| Column | Notes |
|--------|--------|
| `favorites` | Semicolon-separated task names (comma allowed if no semicolons). Max **3** entries after normalization. |
| `dislikes` | Same format and limit. No task in both lists; names must exist in the **Task Catalog** at import time. |

Row errors report **spreadsheet row** numbers (header = row 1).

## 3. Task catalog (not a CSV contract)

Assignable tasks and default headcounts live in the **`tasks`** table, edited in the **Task Catalog** UI (`task_repository.py`). Technician favorites/dislikes are validated against **catalog task names**.

Illustrative YAML (not loaded by the app): [examples/sample_task_list.yaml](examples/sample_task_list.yaml).

## 4. Exported assignment CSVs (output)

These are **generated by the UI** (pandas `to_csv`), not re-imported by the app today.

### Draft download (Home, after generate)

Typical columns: `task`, `technician` (display name), `tech_id`.

### Published slice download (post-publish banner or history)

Confirmed rows use dict keys from `assignment_repository.load_confirmed_assignment_rows_for_slice` / `load_confirmed_assignment_rows_for_date`:

| Column | Notes |
|--------|--------|
| `work_date` | Date |
| `time_slot` | `AM`, `MID`, or `PM` |
| `tech_id` | FK to `technicians.tech_id` |
| `tech_name` | From join; falls back to `tech_id` if profile missing |
| `task` | Stored task label (`assignments.task_id` â€” task **name** string in current schema) |
